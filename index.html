Private messenger built on Mantle-compatible smart contracts

Encrypted chats. Minimal public metadata.

Mantle Private Messenger is a browser-first encrypted messenger where users own their identity, write message state through their own on-chain account layer, exchange encrypted invitations through one shared main contract, and keep readable content client-side only.

Architecture

One public protocol, many private local views.

The protocol separates identity, invitations, per-user writes, encrypted message state and encrypted file storage. The blockchain provides ordering and availability. The browser reconstructs the readable messenger locally.

Alice browser

Holds Alice wallet key for transactions and Alice messenger keys for decrypting invitations.

Alice local DB

Stores decrypted chats, cached files, local profile state and local signer accounts.

MainConnector

The shared entry point. It stores user registry data and a mixed stream of encrypted invitation envelopes. Observers see ciphertext blobs, not readable chat names or invitation content.

User account layer

After a user accepts or creates a chat, actual encrypted chat activity is written through the user's own account path instead of one global public room.

Bob browser

Scans mixed invitations, tries to decrypt only envelopes intended for Bob, ignores the rest.

Encrypted IPFS files

Attachments are encrypted before upload. IPFS sees bytes and CIDs, not file plaintext.

Encryption flow

Readable content exists only after local decryption.

The messenger uses two different key layers: an EVM signing key for transactions and a messenger encryption layer for private communication. They are intentionally separate.

Chat key created locally

The browser creates a symmetric chat key. This key is never published in plaintext.

Invitation encrypted

The chat key and invitation metadata are encrypted for the recipient. Only the recipient can open it locally.

Messages encrypted

Message text is encrypted before it touches contracts, logs, indexers or IPFS.

Files encrypted before IPFS

The app uploads encrypted attachment bytes. The message stores only the encrypted file reference and IV.

Important separation

Wallet key signs transactions. Messenger key decrypts communication.

The EVM private key pays gas and proves account ownership. It is not used as the file or message encryption key. Messenger encryption keys are used to open invitations and decrypt chat state inside the browser.

Local first

State is reconstructed locally

The chain gives encrypted events. The browser syncer reads them, decrypts what it can, and builds the user's local inbox.

Metadata model

Metadata is reduced by design, not magically erased.

The protocol avoids a single public chat room model. It mixes encrypted invitations in the main contract and pushes encrypted conversation activity into user-specific account paths.

Hidden or minimized

What observers should not read directly

✓ Plain message text is not stored on-chain.
✓ Plain file content is not uploaded to IPFS.
✓ Invitation payloads are mixed as encrypted envelopes in the shared MainConnector.
✓ Recipients discover only invitations that their local messenger key can decrypt.
✓ Each user writes encrypted chat state through their own account flow, reducing global chat graph leakage.
Still public

What blockchains still reveal

! Transactions, sender addresses, gas usage and timestamps remain visible.
! IPFS CIDs and encrypted file sizes can exist as metadata.
! Repeated interaction timing can still leak behavioral patterns.
! This is private messaging architecture, not full anonymity infrastructure.

AI agent integration

Agents can use the messenger through a restricted core wrapper.

Mantle Private Messenger is designed so AI agents do not need raw unrestricted wallet or contract access. Instead, an agent can be connected through a small dedicated package that wraps the messenger core and exposes only approved read and write operations.

Agent core

@mantle/messenger-ai-agent-core

The AI-agent package sits on top of @mantle/messenger-core. It reuses the same sync, encryption, local database and contract write logic, but presents a narrower API made for controlled automation instead of full application access.

No raw wallet API

Controlled signing

The agent can be given a private-key based signer or a delegated local signer, but the public API does not expose arbitrary wallet clients or arbitrary contract calls.

Same privacy model

Local decryption

Agents read the same locally reconstructed messenger state. Plaintext still appears only after the owner gives the agent access to the required local context and keys.

User-controlled context

The owner decides which account, signer, chat state and local messenger keys are available to the agent.

Policy boundary

The agent is not the protocol owner. It receives a constrained interface with specific allowed actions.

AI Agent Core

A restricted client layer that exposes whitelisted read/write functions and delegates real protocol work to the messenger core.

Messenger Core

The same core package powers the web app and the agent integration: encryption helpers, IndexedDB state, syncer, IPFS attachment logic and contract write actions.

Smart contracts

The agent writes through the same protocol paths as a normal user: registry, mixed invitations and user-account message flows.

Encrypted network state

Contracts and IPFS still receive encrypted payloads. The agent does not make plaintext public.

Read safely

An agent can inspect local chats, messages, invitations, registration status and health checks through a limited read API.

Draft or summarize

The agent can summarize unread conversations, prepare replies, explain attachments or help manage incoming invitations.

Write with permission

Allowed writes can include registering the agent account, creating chats, sending messages, inviting users and accepting invitations.

Keep the boundary

The restricted wrapper avoids turning an AI assistant into a fully unrestricted wallet, contract caller or raw database editor.

Good agent tasks

What the agent layer is useful for

✓ Summarize recent encrypted conversations after local decryption.
✓ Prepare replies and ask the user before sending.
✓ Watch for new invitations and explain who can be accepted.
✓ Operate a service-style messenger account with a controlled private key.
Intentional limits

What it should not expose by default

! No arbitrary contract calls from the agent API.
! No raw unrestricted wallet client as the public interface.
! No silent access to user plaintext without the user-provided local context.
! No assumption that automation equals ownership. The user remains the control point.

Protocol summary

Public availability. Private readability.

Mantle Private Messenger treats the blockchain as an encrypted coordination layer. Contracts make user identities, encrypted invitation envelopes and encrypted event ordering available. The browser makes them readable only for the user who owns the right local keys.
